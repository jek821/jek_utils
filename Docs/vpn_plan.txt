jekvpn roadmap
==============

Goal: build a small mesh VPN with a coordination server (jekvpn) --- a
miniature, self-hosted Tailscale/headscale. Each tool below teaches a skill
that jekvpn directly depends on.

Two planes, like the real thing:
  - DATA PLANE  (the WireGuard analog): encrypted peer-to-peer UDP tunnels
                that carry the actual traffic. The coordinator never sees it.
  - CONTROL PLANE (the headscale analog): a server nodes register with. It
                hands each node a "peer map" --- every other node's public key,
                virtual IP, and real endpoint (IP:port). This is what headscale
                IS. Understanding it is the point of the project.


--- SHARED TRUNK (lives in libjek --- link it, don't rebuild) ---

  jekvpn is a SECOND capstone hanging off the same ladder as jeksync. The
  control plane is a wire protocol over sockets served by a concurrent daemon
  --- skills sync_plan.txt already builds. These rungs, once a second consumer
  needs them, get extracted into libjek/ (see libjek/README.md) and BOTH
  capstones link the same module -- the shared logic is written once, not
  reimplemented per tool. Reuse directly:

  jekcrypto  (sync Phase 6) -- libsodium keygen, nonces, AEAD.
                               DESIGN SWAP (this is the whole ballgame): do NOT
                               encrypt each packet directly to a peer's static
                               public key with crypto_box. That naive design has
                               no forward secrecy, no replay protection, and
                               reuses nonces on restart. Instead the data plane
                               runs a NOISE session (Noise_IK -- what WireGuard
                               uses): a handshake establishes EPHEMERAL session
                               keys, then packets are AEAD-sealed under those
                               keys with a counter nonce + sliding replay window.
                               This one swap resolves forward secrecy, replay,
                               rekeying, nonce uniqueness, session protocol, and
                               peer identity at once. DO NOT ROLL YOUR OWN
                               handshake -- adopting Noise is the single change
                               that separates "real" from "leaks silently".
  jekproto   (sync Phase 8) -- length-prefixed, typed binary messages. The
                               control protocol IS jekproto: message types
                               REGISTER, ASSIGN_IP, PEER_MAP, PEER_UP/DOWN.
  jeksend /
  jekrecv    (sync Phase 9) -- socket/connect (node->coordinator client) and
                               bind/listen/accept (coordinator server).
  jekpar     (sync Phase 7) -- pthreads/mutex/work queue: the coordinator
                               serving many nodes at once.
  jeksig     (sync Phase 7) -- SIGTERM/graceful shutdown: both daemons need it.

  ~70% of jekvpn is the trunk above. Only the data plane below is new.


--- DATA PLANE (C) --- new rungs ---

PHASE V1 - Virtual interface
  jektun  -- open /dev/net/tun, ioctl(TUNSETIFF) to create a tun device,
             bring it up, read/write raw IP packets. Parse the IP header
             (src/dest address) of outgoing packets.
             ("jekcat for a network interface" --- the one truly new
              low-level skill. Needs sudo / CAP_NET_ADMIN.)

PHASE V2 - Datagram I/O + event loop
  jekmux  -- UDP datagram sockets (sendto/recvfrom) and poll() to drive TWO
             fds at once: the tun device and the UDP socket.
             (sync's networking rungs are TCP streams; the data plane is
              connectionless datagrams plus your first real event loop.)


--- CONTROL PLANE + CAPSTONE (C) ---

PHASE V3 - The coordinator (the headscale analog)
  jekcoord -- a server that:
                - accepts node registrations (jekproto REGISTER over jekrecv)
                - PROVES KEY OWNERSHIP at registration: challenge the node with a
                  nonce, node signs it, coordinator verifies against the pubkey
                  it presents. Without this, anyone registers as anyone. The
                  control channel itself is authenticated + encrypted (Noise or
                  TLS), never plaintext.
                - assigns each node a virtual IP from a pool (e.g. 10.99.0.0/24)
                - records each node's public key + real endpoint (learned from
                  the source address of its packets --- poor-man's STUN, exactly
                  how Tailscale discovers endpoints)
                - pushes every node an updated PEER_MAP on join/leave
              = jekproto served over jekrecv + jekpar (many nodes) + jeksig.
              This binary is, in miniature, headscale.

PHASE V4 - The node + composition
  jeknode -- the client daemon. Composes the whole ladder:
                jektun      read/write the tunnel
              + jekmux      poll the tun fd and UDP socket; route by dest IP
              + jekcrypto   run a Noise session per peer; AEAD-seal each packet
              + jekproto /
                jeksend     register with jekcoord, receive the PEER_MAP
              + jeksig      clean shutdown, tear down the tun device

             MODEL: cryptokey routing (WireGuard's core idea). A peer's public
             key is authorized for a specific set of virtual IPs, and that
             mapping IS simultaneously the routing table AND the access-control
             list. This framing is what makes the source-IP check below natural.

             Core loop:
               tun  -> read IP packet -> look up dest virtual IP -> find the
                    peer authorized for it -> seal under that Noise session
                    -> sendto its endpoint
               sock -> recvfrom -> open with the session that decrypts it
                    -> VALIDATE: inner IP packet's SOURCE must equal the virtual
                       IP bound to that peer's key (else a valid peer is spoofing
                       another node -- drop it) -> write packet to tun

PHASE V5 - Connectivity: relay-first, then hole-punch (revised)
  jekrelay -- build the RELAY FALLBACK FIRST: a DERP-style encrypted relay so
              every peer PAIR has a working path from day one. "Works across the
              internet" instead of "works on my LAN."
  jekhole  -- THEN add UDP hole punching (coordinated by jekcoord) purely as a
              latency optimization for pairs that can connect directly. You do
              not need to solve the company-sized hole-punching problem to have
              a real, usable VPN -- relay covers everyone; direct just improves
              the ones it can.
  keepalive + mobility -- persistent keepalive to hold NAT mappings open;
              re-learn a peer's endpoint when its source address changes
              (roaming / NAT rebind), on an AUTHENTICATED packet only.

PHASE V6 - Hardening (NOT optional --- this is where toy becomes real)
  never trust the wire -- reject malformed / oversized packets; handle
                          reordering, duplication, loss (Noise replay window
                          already covers duplicates/too-old).
  rekey                 -- rotate session keys on time and on data volume.
  anti-DoS              -- rate-limit handshakes; cap concurrent sessions;
                          bound per-peer memory so a flood can't exhaust the box.
  key storage           -- private keys stored securely, not world-readable.
  MTU                   -- account for tunnel overhead so large packets don't
                          silently fragment or vanish.


--- TESTING ---
  No second machine needed. Linux network namespaces (ip netns) stand up
  several isolated "hosts" on one box; run jekcoord in one and a jeknode in
  each of the others to exercise a full mesh locally, in the test.sh style.


--- shared vs new ---
  shared via libjek         : jekcrypto, jekproto, jeksend, jekrecv,
                              jekpar, jeksig
  new for jekvpn            : jektun, jekmux, jekcoord, jeknode
                              (+ jekrelay, jekhole)


--- SCOPE: what "legitimate" means here ---
  Target: a secure mesh between YOUR OWN machines. Reachable solo and honestly
  usable. Out of scope (and out of any claim): protecting other people's
  traffic, or standing up to a serious adversary. The linchpin decision that
  keeps "legitimate" achievable rather than dangerous: adopt Noise, never roll
  your own crypto.
